**Duration:** 60 min build + 60 min optional rehearsal + office hours · **Theme:** The founder's eye.

## The one idea you need

You've been staring at your project all week. You've stopped seeing it.

The **founder's eye** is what happens when you watch someone use your thing for the first time and see it the way they do — confused by things you thought were obvious, annoyed by things you thought were fine.

Today you get that perspective on purpose, before Demo Day, when you can still fix things.

---

## The 5 things first-time users hit

**1. The first 5 seconds**
What does a user see before they understand what your tool does? If it's a blank input box with no explanation — they'll close it.

**2. Error states**
What happens when the AI gives a weird response? What if the input is empty? What if something breaks? "Undefined" is not a helpful error message.

**3. Dead ends**
Is there anywhere a user can get stuck with no way forward? A button that does nothing? A result with no "try again" option?

**4. Copy tone**
Is your button text consistent with your Day 3 tone choice? Check every label, placeholder, and error message.

**5. Load time**
If your AI call takes 5+ seconds, the user needs feedback — a spinner, a "thinking..." message, something. Silence feels like it broke.

## Build steps

**Step 1 — Write your 3 friction points (10 min)**

Open your deployed app fresh — pretend you've never seen it before.

Find 3 things that would confuse, annoy, or lose a first-time user. Write them in Discord:

```
Friction 1: [what's wrong + where]
Friction 2: [what's wrong + where]
Friction 3: [what's wrong + where]
```

Be honest. The more specific you are, the faster you fix it.

> ✅ If you can't find 3 — share your screen in Discord and someone will find them for you in 60 seconds.

**Step 2 — Fix all 3 (30 min)**

Tell Replit Agent each fix one at a time:

> "Fix friction point 1: [describe exactly what's wrong and what should happen instead]"

Review each fix before moving to the next one. Don't let Agent fix all 3 in one shot — review each one.

Common fixes that always help:
- Add a 1-sentence description at the top explaining what the tool does
- Add a loading indicator while the AI is thinking
- Add example input so users know what to type
- Fix any error messages to say something human instead of a code error
- Make the submit button label match your tool's action ("Generate quiz" not "Submit")

**Step 3 — Lock your 60-second demo path**

Write down your exact demo script:

```
1. Hook (10s): "I built [name]. It does [one thing] for [who]."
2. Demo (40s): [exact input you'll type] → [what the output looks like] → [the moment that shows why it's useful]
3. So-what (10s): "The problem this solves is ___. I'd keep building it to ___."
```

Rules:
- Start with the thing **working** — not a loading screen or empty state
- Show **real input, real output** — don't narrate what it would do
- If the live demo could break → have a backup screenshot ready
- If it takes more than 10 seconds to show the first output → cut that part of the demo

**Step 4 — Final GitHub push**

Commit message: `"Day 7: final polish — 3 friction fixes, demo-ready"`

> ✅ Ship proof: final GitHub push link + your 3 friction fixes posted in `#day-7-ships`.

## Optional office hours (60 min after the main session)

**Demo rehearsal pairs:**
- Partner up in a breakout room
- Each person runs their 60-second demo while the other times it
- Partner gives feedback: "I understood" / "I was confused by ___" / "The best part was ___"
- Two rounds each

Then: Replit deployment help, final GitHub push, README last polish.
